# Code Review — модуль `Shared\Deletion` (полный разбор)

Ревью фрагмента внутренней библиотеки удаления сущностей (PHP 8.1+, Symfony + Doctrine ORM).
Разбор охватывает **все файлы проекта**: атрибут, enum'ы, DTO, middleware, сервис анализа и оркестратор,
а также README. Ссылки на строки даны по версии кода на момент ревью.

Модуль решает две задачи:

- **Анализ** связей сущности через PHP-атрибуты `RelationTo` — `DeletionService` (что будет затронуто,
  можно ли удалять).
- **Исполнение** каскадного удаления с хуками-middleware внутри транзакции — `DeletionOrchestrator`.

## 1. Общая оценка

Идея модуля здравая: декларативное описание связей рядом с сущностью и чистое разделение
«анализ / исполнение». Код современный (enum, `readonly`, атрибуты, named arguments), в целом читаемый.

Однако есть системная проблема: **заявленный контракт расходится с фактическим поведением**.
Это видно на трёх уровнях сразу — README описывает несуществующий атрибут; `analyze()` содержит мёртвую
ветку блокировки; а `DeletionOrchestrator::execute()` вообще не сверяется с результатом анализа и удаляет
то, что `canDelete` пометил неудаляемым. Плюс в исполняющей части — отсутствие защиты от циклов, N+1,
нетопологический порядок удаления и несколько латентных багов на нестандартных идентификаторах.

**Вывод:** аналитическая часть (`DeletionService`) близка к production после правки нескольких edge cases;
исполняющая (`DeletionOrchestrator`) требует доработки перед боевым использованием — в текущем виде она
может как «недоудалить» (нарушить FK), так и «переудалить» (проигнорировать блокировку), а на циклических
графах — уйти в бесконечную рекурсию.

Оценка по областям:

| Область | Оценка | Комментарий |
|---|---|---|
| Архитектура (макро) | 🟢 хорошо | Правильные границы, расширяемость через middleware |
| Аналитика (`DeletionService`) | 🟡 удовлетворительно | Работает, но edge cases и мёртвая логика |
| Исполнение (`DeletionOrchestrator`) | 🔴 требует доработки | Критичные баги корректности и производительности |
| Типобезопасность / модель | 🟡 удовлетворительно | Позиционные массивы, слабо типизированные «relation»/«plan» |
| Документация | 🔴 вводит в заблуждение | README описывает другой (несуществующий) API |
| Тестируемость | 🟢 хорошо | Зависимости инъектируются, легко мокаются (см. добавленные тесты) |

## 3. Архитектурный разбор

### 3.1. Что сделано хорошо

- **Декларативность.** Атрибут `RelationTo` с `Attribute::IS_REPEATABLE` держит знание о связях рядом с
  моделью и позволяет несколько связей на класс. Онбординг новых сущностей — без правки ядра.
- **Разделение «анализ / исполнение».** `DeletionService` можно переиспользовать в UI («покажи, что
  удалится») без риска что-либо удалить; `DeletionOrchestrator` отвечает только за мутации.
- **Расширяемость через middleware.** Фазовые хуки before/after дают точки для логирования, метрик, аудита
  без изменения ядра (OCP).
- **`dryRun`.** Хорошая идея — прогнать план без мутаций (с оговоркой F10).
- **Инъекция зависимостей.** `EntityManagerInterface`, `GenericReadRepository`, `iterable $middlewares` —
  всё через конструктор, что делает модуль тестируемым (подтверждено добавленными unit-тестами).

### 3.2. Ключевые архитектурные замечания

- **Рассинхронизация контракта на трёх уровнях** (F1, F12, F20) — главная системная проблема: документация,
  аналитика и исполнение описывают/реализуют разные представления о том, что такое «блокировка».
- **Дублирование логики обхода связей.** Правила связей читаются и интерпретируются в двух местах:
  `DeletionService` (карта + резолверы) и `DeletionOrchestrator::buildRecursive` (снова разбор правил,
  снова загрузка детей). Это два источника истины про одну модель — риск расхождения.
- **Слабо типизированная модель.** Правила связей передаются позиционным массивом
  `array{0:.., 1:.., ...}` (`DeletionService.php:17`), а план удаления — массивом ассоциативных массивов
  (`OrderedPlanDto`). Напрашиваются value-object'ы `RelationRule` и `DeletionStep`.
- **Смешение ответственностей в `DeletionService`** (SRP): скан атрибутов, работа с метаданными, парсинг
  JSON, запросы к join-таблицам и чтение FK через Reflection — в одном классе.

## 4. Пофайловый разбор

### 4.1. `Attribute/RelationTo.php`

**Хорошо:** компактный immutable-атрибут, `IS_REPEATABLE`, разумные значения по умолчанию.

**Замечания:**

- **F16 — нет валидации сочетаний параметров.** Конструктор принимает любые комбинации. Например,
  `joinTable` без `joinColumn`/`inverseJoinColumn` (или наоборот) молча приведёт к неверной ветке в
  резолвере; сочетание `type: BLOCKING` + `cascade: DELETE_CHILD` семантически противоречиво (одновременно
  «нельзя удалять родителя» и «удалить ребёнка вместе с родителем»), но не отвергается. Стоит валидировать
  в конструкторе (или через именованные конструкторы `RelationTo::scalar()`, `RelationTo::manyToMany()`).
- **Двусмысленность `field` при M2M.** README сам называет его «формально обязательным» — признак того, что
  модель смешивает два разных вида связи в одном атрибуте. Для M2M `field` не несёт смысла.
- `entity` типизирован как `string` — стоит `class-string` (для статанализа/PHPStan).
- Атрибут смешивает два ортогональных аспекта: *обнаружение связи* (`entity`/`field`/`joinTable...`) и
  *политику удаления* (`type`/`cascade`). Для узкоспециализированной библиотеки удаления допустимо, но
  явное разделение (или доккомментарий про допустимые комбинации) снизило бы риск ошибок.

### 4.2. `Enum/DeletionCascade.php`, `Enum/RelationType.php`

**Хорошо:** string-backed enum'ы с понятными комментариями вместо «магических строк».

**Замечания:**

- **Концептуальное перекрытие `type` и `cascade`.** `RelationType` (BLOCKING/REFERENCE) и `DeletionCascade`
  (NONE/DELETE_CHILD/DETACH_RELATIONS) частично пересекаются по смыслу и порождают невалидные комбинации
  (см. F16). Возможная альтернатива — единый enum политики связи
  (`BLOCK` / `CASCADE_DELETE` / `DETACH` / `IGNORE`), исключающий противоречия по построению.
- В `DeletionService` `RelationType` немедленно сводится к `bool $isBlocking` — сам enum дальше не живёт;
  это усиливает подозрение, что двух enum'ов больше, чем нужно.

### 4.3. `Dto/CanDeleteDto.php`

**Хорошо:** `final readonly`, минимальный контракт результата.

**Замечания:** порядок аргументов в докблоке (`@param ... $dependents`, затем `$canDelete`) обратен
сигнатуре (`canDelete`, `dependents`) — косметика, но сбивает.

### 4.4. `Dto/DependentGroupDto.php`

**Хорошо:** иммутабельная группа зависимостей.

**Замечания:**

- **F21 — `count` дублирует `count($ids)`.** Хранение обоих полей допускает рассинхронизацию (в коде их
  действительно заполняют раздельно). Лучше вычислять `count()` из `ids` (или гарантировать инвариант).
- Поле `hard` для «родительских» групп всегда `false` (см. F12) — семантика поля зависит от контекста
  (родитель/ребёнок), что не выражено типом. Стоит развести на два DTO или задокументировать.

### 4.5. `Dto/OrderedPlanDto.php`, `Dto/RelationsDto.php`

**Замечания (F19):**

- Несогласованность стиля: `OrderedPlanDto`/`RelationsDto` — `final class` с `readonly`-полями, тогда как
  `CanDeleteDto`/`DependentGroupDto` — `final readonly class`. Стоит привести к одному стилю.
- В `OrderedPlanDto` докблок малформед — два `@param` в одной строке (`Dto/OrderedPlanDto.php:10`).
- `delete`/`detach` — слабо типизированные `array` из ассоциативных массивов. Value-object'ы
  (`DeleteStep`, `DetachStep`) убрали бы «строковые ключи» и сделали план самодокументируемым.

### 4.6. `Middleware/DeletionMiddlewareInterface.php`

**Хорошо:** явные фазовые хуки, покрывающие жизненный цикл удаления.

**Замечания:**

- **F11 — `supports()` не вызывается** оркестратором (используется `method_exists`). Либо соблюдать
  контракт (`if ($mw->supports($root::class))` перед `notify`), либо убрать метод. Сейчас реализующий
  класс вводится в заблуждение.
- **«Толстый» интерфейс (ISP).** Любой middleware обязан реализовать все 6 хуков, даже если ему нужен один.
  И `LoggingDeletionMiddleware`, и добавленный `MetricsDeletionMiddleware` вынуждены реализовывать полный
  набор. Полезны были бы: базовый no-op трейт/абстракция или дробление на под-интерфейсы
  (`DetachAware`, `DeleteChildrenAware`, `DeleteRootAware`).
- `array $relation` — не типизирован; напрашивается DTO вместо ассоциативного массива с
  `joinTable`/`joinColumn`/... .
- Докблоки перечисляют параметры в порядке, отличном от сигнатуры (косметика).

### 4.7. `Middleware/LoggingDeletionMiddleware.php`

**Хорошо:** корректная инъекция PSR-3 `LoggerInterface`, ясные сообщения.

**Замечания:**

- `supports()` всегда `true` — ожидаемо (F11).
- **Уровень и объём логов.** `info` с полными `childIds`/`relation` на массовых операциях может генерировать
  большой объём и логировать тысячи id; рассмотреть `debug` и/или логировать `count($childIds)` вместо
  полного списка.
- **Непоследовательный контекст.** В detach/children логируется `parentClass`/`childClass`, но не `$root`;
  в root — только класс. Единый контекст (root id/class на всех фазах) упростил бы трассировку.

### 4.8. `DeletionService.php`

**Хорошо:** ядро аналитики; чистые публичные методы `canDelete()`/`analyze()`; кэш карты и метаданных.

**Замечания:**

- **F12 — мёртвая ветка `hasHardParent`** (`:47`). `findParentsByAttributes` всегда ставит
  `isBlocking = false` (`:122`, концептуально верно — BLOCKING на ребёнке блокирует *родителя*), поэтому
  условие `if ($group->hard && ...)` не срабатывает никогда. Блок и поле `hard` у «родительских» групп —
  мёртвые/вводящие в заблуждение. Убрать проверку или явно задокументировать «родитель — всегда инфо».
- **F3 — `getJoinTableParentIds()`** (`:216`) строит DQL `->from($joinTable, 'jt')` по имени M2M-таблицы,
  а `from()` ждёт FQCN сущности → на реальном M2M DQL упадёт при парсинге. Нужен native SQL/DBAL (как в
  `DeletionOrchestrator::detachJoinRow`).
- **F4 — `getScalarFkValue()`** (`:227`) читает свойство-ассоциацию (`private OrderEntity $order`) через
  Reflection и трактует как `int|string` → в `ids` попадёт объект/Doctrine-proxy, а не id. Нужно различать
  ассоциацию и скаляр по метаданным Doctrine и брать id связанной сущности.
- **F5 — `getValue()` на неинициализированном typed-property бросает `Error`** (`:235`). Если сущность
  имеет `private int $parentId;` без значения по умолчанию и оно не инициализировано, Reflection выбросит
  `Error: ... must not be accessed before initialization`. Нужна проверка `isInitialized($object)`.
- **F13 — JSON edge cases** (`getJsonArrayParentIds`, `:173`): `if (empty($jsonValue))` (`:183`) отбрасывает
  валидный id `0`/`'0'`; предикат `is_numeric($id) || is_string($id)` — почти no-op; `array_filter`
  сохраняет ключи → результат перестаёт быть `list`, хотя DTO обещает `list<int|string>`. Проверять
  `null`/не-массив явно и переиндексировать через `array_values`.
- **F15 — `ensureMap()`** (`:82`) вызывает `getMetadataFactory()->getAllMetadata()` — прогрев метаданных
  ВСЕХ сущностей приложения на первом обращении. Дорого на «холодном» контейнере. Стоит кэшировать готовую
  карту (PSR-6) или собирать её на этапе компиляции DI.
- **F17 — `NONE + REFERENCE` дети запрашиваются, но не используются** (`findChildrenByAttributes`, ветки на
  `:284`). При `cascade=NONE` и `type=REFERENCE` (не hard) дети находятся (запрос выполнен), но не попадают
  ни в `childrenDelete`, ни в `childrenDetach` — результат отбрасывается. Лишняя нагрузка на БД; либо не
  запрашивать такие связи, либо возвращать их как информационные.
- **Дублирование Reflection на горячем пути.** В `findParentsByAttributes` на каждый атрибут создаётся до
  трёх `new ReflectionClass()` (сам метод + `getJsonArrayParentIds` + `getScalarFkValue`). Кэшировать по классу.
- **F18 — `setAccessible(true)`** (`:180`, `:235`) — no-op с PHP 8.1, `@deprecated` с 8.5 (подтверждено
  прогоном тестов). Вызовы можно удалить.
- **Инкапсуляция.** `getChildRelationRules()` (`:306`) публично отдаёт внутренний позиционный кортеж —
  оркестратор завязан на индексы `[0..6]`. Отдавать `RelationRule[]`.
- **Версионная связанность с Doctrine.** `isJsonField()` (`:313`) читает `$fieldMapping['type']` как массив
  — это ORM 2.x; в ORM 3 маппинги стали объектами. При апгрейде сломается.

### 4.9. `Service/DeletionOrchestrator.php`

**Хорошо:** транзакционность (`wrapInTransaction`), фазовый порядок (detach → children → root), `dryRun`,
корректный native SQL в `detachJoinRow`.

**Замечания:**

- **F1 — `execute()` не проверяет `canDelete`** (`:28`). Строит план и удаляет, игнорируя
  `analyze()->canDelete`. «BLOCKING»-связь не блокирует реальное удаление. Нужна проверка в начале
  (`throw new DeletionBlockedException`) или явный `force`.
- **F2 — нет visited-set в `buildRecursive()`** (`:89`). Циклическая связь (A→B→A) → бесконечная рекурсия;
  диамант → повторный обход. `deleteMap` дедуплицирует id для удаления, но не защищает обход. Нужен
  `array<string,bool> $visited` по ключу `class#id`.
- **F8 — N+1** (`:124`). Для каждого id ребёнка — отдельный `repository->find()`/`findOneBy()` и затем
  `analyze()`. На большом дереве — тысячи запросов. Нужна батч-загрузка (`findBy(['id' => $ids])`).
- **F9 — удаление по неверной колонке при нестандартном PK** (`:112`). `idField` выбирается как `'id'`,
  если `'id'` входит в идентификаторы, иначе — `group->field` (FK на родителя). Затем `deleteByIds`
  выполняет `DELETE ... WHERE e.<idField> IN (:ids)`, где `:ids` — это id самих детей. Для сущностей без
  поля `id` это удаление по колонке FK значениями, которые являются id детей → удалит не те строки (или
  ничего). Латентный баг корректности; нужно единообразно удалять по идентификатору сущности.
- **F7 — нетопологический порядок** (`buildOrderedPlan`, `:67`). `deleteMap` группирует по классу в порядке
  вставки (сверху вниз). При FK между уровнями bulk-DELETE может нарушить целостность. Нужен обход/удаление
  снизу вверх (сначала листья).
- **F6 — `notify()` глушит все исключения** (`:156`). Внутри транзакции удаления `catch (Throwable) {}`
  проглатывает любую ошибку middleware (в т.ч. симптом реального бага) без следа. Дифференцировать политику:
  «наблюдателей» глушить, но логировать факт; критичные — прокидывать.
- **F10 — `dryRun` не доходит до middleware** (`:33`). В `execute` при `dryRun=true` мутации пропускаются,
  но `notify()` вызывается всегда — метрики/логи (включая добавленный `MetricsDeletionMiddleware`) посчитают
  dry-run как реальное удаление. Хукам нужен флаг `dryRun` (или отдельные хуки).
- **F14 — bulk DQL DELETE** (`deleteByIds`, `:180`) не поднимает lifecycle-события Doctrine
  (`preRemove`/`postRemove`), игнорирует `orphanRemoval` и ORM-каскады — тихое расхождение с `remove()`.
  Осознанно выбрать и задокументировать стратегию.
- **F22 — `detachJoinRow`** (`:168`): имена таблицы/колонок подставляются в SQL через `sprintf` без
  квотирования идентификаторов (риск на зарезервированных словах; инъекция маловероятна, т.к. значения из
  атрибутов, а не из пользовательского ввода). Плюс нет защиты от пустого `childIds` → `IN ()` даст
  невалидный SQL (сейчас защищено косвенно тем, что detach добавляется только при непустых ids).
- **Композитные идентификаторы.** `array_values($parentIdArr)[0] ?? null` (`:92`) берёт только первую
  колонку — составные PK не поддержаны.
- **`$relations->parents` не используется** в исполнении — оркестратор оперирует только
  `childrenDelete`/`childrenDetach`.
- Докблок конструктора (`:15`) перечисляет параметры в порядке `middlewares, em, analyzer`, а сигнатура —
  `em, analyzer, middlewares`. Косметика.

### 4.10. `README.md`

**F20 — документация описывает другой API.** Раздел «3. Параметры атрибута DependsOn» и часть примеров
описывают атрибут **`DependsOn`** с параметрами `parent`/`hard`, тогда как в коде — **`RelationTo`** с
`entity`/`type`/`cascade`. При этом пример в разделе 1 уже использует `RelationTo` — README противоречит
сам себе. Пример из раздела 3 не скомпилируется. Также описана консольная команда `deletion:check`,
класс которой в переданный фрагмент не входит (проверить наличие в полном проекте). README нужно привести
в соответствие с кодом.

## 5. Сквозные темы

### 5.1. Производительность

- **N+1 при планировании** (F8) — главный риск на больших деревьях.
- **Прогрев всех метаданных** (F15) на первом вызове анализа.
- **Лишние запросы** для `NONE+REFERENCE` детей (F17) и загрузка полных сущностей ради одного id
  (`findByAssociation` + `getId` вместо выборки только id).
- **Middleware внутри транзакции** — медленный сток метрик/логов удлиняет удержание блокировок БД;
  тяжёлые сайд-эффекты лучше выполнять после коммита.
- **Дублирующий кэш метаданных** (`$metadataCache`) поверх кэша Doctrine.

### 5.2. Корректность и надёжность

- Игнор блокировки (F1), бесконечная рекурсия (F2), неверная колонка удаления (F9), нетопологический
  порядок (F7), `Error` на неинициализированных свойствах (F5), падение DQL на M2M (F3) — это дефекты,
  проявляющиеся на вполне обычных конфигурациях (нестандартный PK, циклические/диамантовые графы, M2M).
- Проглатывание исключений (F6) маскирует все вышеперечисленное в проде.

### 5.3. Безопасность

- Риск SQL-инъекции низкий: имена таблиц/колонок берутся из атрибутов (dev-controlled), значения — через
  плейсхолдеры. Тем не менее идентификаторы стоит квотировать (F22), а конфигурацию — валидировать (F16).

### 5.4. SOLID / PSR / Symfony

- **SRP** — `DeletionService` совмещает слишком много (см. 4.8); напрашиваются `RelationMapBuilder`,
  `RelationResolver`, `RelationAnalyzer`.
- **OCP/типобезопасность** — позиционные массивы правил и «строковые» планы вместо VO.
- **ISP** — «толстый» `DeletionMiddlewareInterface` + неиспользуемый `supports()` (F11).
- **DIP** — прямая зависимость от конкретного `GenericReadRepository` вместо узкого интерфейса-читалки.
- **PSR-3** — используется корректно в логировании; тот же логгер стоит задействовать при потере ошибок в
  `notify()`.
- **Консистентность** — стиль DTO (F19), докблоки, README (F20).

### 5.5. Тестируемость

Плюс модуля: зависимости инъектируются, поведение легко изолируется моками (подтверждено добавленными
unit-тестами `DeletionService` — покрыты blocking/DELETE_CHILD/DETACH/NONE, parent-ref, JSON, FK=0,
множественные `RelationTo`). Тесты заодно фиксируют текущее (в т.ч. спорное) поведение как «executable
specification» — при исправлении багов такой тест подскажет, что контракт поменялся.

## 6. Рекомендации и порядок правок

Приоритезированный список (сначала — то, что влияет на корректность в проде):

1. **Сверять `canDelete` в `execute()`** (F1) — восстановить контракт блокировки.
2. **Ввести visited-set и топологический порядок** в планировщике (F2, F7).
3. **Исправить удаление по идентификатору** для нестандартных PK (F9).
4. **Native SQL для join-таблиц в анализе** (F3), как уже сделано в `detachJoinRow`.
5. **Различать ассоциацию и скаляр** и обрабатывать неинициализированные свойства (F4, F5).
6. **Явная политика ошибок middleware** + логирование факта потери (F6); прокинуть `dryRun` в хуки (F10).
7. **Батч-загрузка детей** и кэш карты связей (F8, F15); не запрашивать неиспользуемые связи (F17).
8. **Рефакторинг модели:** `RelationRule`/`DeletionStep` VO, вызвать/убрать `supports()`, разбить
   `DeletionService` по SRP (F11, типобезопасность).
9. **Причесать edge cases и стиль:** JSON (F13), удалить `setAccessible` (F18), унифицировать DTO (F19),
   валидация атрибута (F16), квотирование идентификаторов (F22).
10. **Синхронизировать README с кодом** (F20).

## 7. Итог

Модуль построен на правильной архитектурной идее и хорошо тестируется, но в текущем виде **не готов к
боевому удалению данных**: исполняющая часть содержит критичные дефекты корректности (F1, F2, F9) и
производительности (F8), а расхождение контракта между документацией, аналитикой и оркестратором (F1, F12,
F20) затрудняет сопровождение. Аналитическая часть после правки нескольких edge cases (F3–F5, F13) вполне
пригодна для сценария «показать пользователю, что удалится».

Рекомендованный минимум перед продом: пункты 1–5 из раздела 6. Остальное — планомерное улучшение
типобезопасности, производительности и документации.
